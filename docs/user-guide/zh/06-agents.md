# AI 代理

> 构建能够按照调度策略或事件驱动，自主监控、分析和处理 IoT 数据的智能代理。

---

## 前置条件

创建 AI 代理之前，请确保已完成以下准备：

- 至少配置并测试过一个 **LLM 后端**，详见 [基础设置](./02-settings.md)
- （可选）已连接设备并正常上报遥测数据
- （可选）已安装扩展以获取额外数据源

代理也可以在没有设备或扩展的情况下运行 —— 使用 **自由模式** 即可处理面向整个平台的通用任务。

---

## 1. 打开代理页面

在主导航中点击 **代理** 进入。页面顶部有三个标签页：

| 标签页 | 用途 |
|-------|------|
| **代理** | 所有已配置的代理列表 |
| **记忆** | 跨代理共享的全局记忆库 |
| **技能** | 指导代理执行操作的技能指南 |

![代理列表（空）](../../img/agents-list-empty.png)

> ① 三个主标签页 —— 代理、记忆、技能。② **创建 AI 代理** 按钮打开全屏代理构建器。③ 尚未创建代理时，页面显示空状态引导。

---

## 2. 创建代理

点击 **创建 AI 代理** 打开全屏代理构建器对话框。

![创建代理对话框](../../img/agent-create-dialog.png)

> ① **基本信息** 区域 —— 名称、描述和系统提示词。② **LLM 后端** 选择器 —— 从已配置的后端中选择。③ **执行模式** —— 聚焦、自由、对话或响应式。④ **调度** —— 定时、cron 或事件驱动。⑤ **资源绑定** —— 绑定设备、扩展或指标。⑥ **创建** 按钮保存代理。

### 2.1 基本信息

| 字段 | 必填 | 说明 |
|------|------|------|
| **名称** | 是 | 显示名称（例如"温室监控助手"） |
| **描述** | 否 | 简要说明代理的用途 |
| **系统提示词** | 是 | 定义代理行为、分析风格和决策准则的核心指令 |
| **LLM 后端** | 是 | 使用的 AI 模型，从设置中已配置的后端中选择 |

**编写有效的系统提示词** —— 好的提示词应包含：

- **角色定义**：「你是一个温室监控代理……」
- **数据上下文**：「你将每 5 分钟接收温度和湿度读数……」
- **决策标准**：「如果温度超过 35°C 持续 10 分钟以上，触发告警……」
- **动作指令**：「告警时需包含当前值、阈值和建议的操作……」
- **输出格式**：「以表格形式呈现分析结果，列包括：指标、数值、状态、建议」

### 2.2 执行模式

选择代理的运行方式：

| 模式 | 工作原理 | 适用场景 |
|------|---------|---------|
| **聚焦** | 仅分析绑定的资源。单次执行，Token 消耗低。 | 监控特定设备、周期性分析、告警 |
| **自由** | 可访问所有平台资源。支持多轮推理（最多 20 次工具迭代）。 | 复杂自动化、探索性分析、跨设备关联 |
| **对话** | 交互式对话模式，实时响应用户消息。 | 对话式助手、按需问答 |
| **响应式** | 绑定事件触发时自动运行（设备数据变化、告警等）。 | 事件驱动自动化、异常即时响应 |

**快速选择指南**：

| 你想要…… | 选择 | 原因 |
|----------|------|------|
| 每 5 分钟监控 3 个传感器 | **聚焦** | 仅访问绑定设备，Token 消耗更低 |
| 让 AI 自由探索所有资源 | **自由** | 可以发现并使用任意资源 |
| 拥有一个交互式 AI 助手 | **对话** | 以对话方式运行，支持自定义指令 |
| 传感器触发时做出响应 | **响应式** | 仅在绑定事件触发时运行 |

### 2.3 调度配置

定义代理何时运行及运行频率：

| 类型 | 配置方式 | 示例 |
|------|---------|------|
| **定时** | 每隔 N 分钟或小时运行 | 每 5 分钟（`300s`）、每小时（`3600s`） |
| **Cron** | 标准 cron 表达式 | 每天上午 9 点（`0 9 * * *`）、工作日上午 8:30（`30 8 * * 1-5`） |
| **事件** | 设备数据变化时触发 | 绑定设备上报新数据时 |

常用 cron 表达式：

| 调度 | 表达式 |
|------|-------|
| 每 5 分钟 | `*/5 * * * *` |
| 每小时整点 | `0 * * * *` |
| 每天上午 9 点 | `0 9 * * *` |
| 工作日上午 8:30 | `30 8 * * 1-5` |
| 每周一上午 10 点 | `0 10 * * 1` |

### 2.4 资源绑定

**聚焦** 模式下，需要绑定代理应分析的具体资源：

| 资源 | 说明 | 示例 |
|------|------|------|
| **设备** | 代理可访问和控制的设备 | 温度传感器、风扇执行器 |
| **扩展** | 需纳入上下文的扩展数据 | 天气预报、目标检测结果 |
| **指标** | 需监控的具体数据字段 | 温度、湿度、气压 |

资源使用 DataSourceId 格式：`{type}:{id}:{field}`

示例：
- `device:greenhouse-sensor-01:temperature` —— 设备指标
- `extension:weather:temp` —— 扩展数据

**自由** 模式下资源绑定为可选项 —— 代理默认可访问所有平台资源。

### 2.5 保存

点击构建器底部的 **创建** 按钮。代理会出现在代理页面中，并按照调度配置开始运行。

## 3. 管理代理

创建代理后，代理页面以卡片形式展示所有代理。

![代理列表](../../img/agents-list.png)

> ① 每张代理卡片显示名称、描述和当前状态。② **开关** 用于启用或禁用代理 —— 禁用后代理停止执行。③ **上次执行** 时间和结果。④ **调度** 信息显示下次运行时间或间隔。

### 代理操作

| 操作 | 方法 |
|------|------|
| **启用/禁用** | 点击代理卡片上的开关 |
| **立即运行** | 点击播放按钮立即触发一次执行 |
| **查看历史** | 点击时钟图标查看所有历史执行记录 |
| **编辑** | 点击铅笔图标修改配置 |
| **删除** | 点击垃圾桶图标并确认（永久删除，无法撤销） |

### 代理详情面板

点击代理卡片主体区域（非操作图标）可打开全屏详情面板，显示：

- **代理信息**：名称、描述、执行模式、调度配置和资源绑定
- **执行历史**：最近运行的时间线，包含状态、耗时和 Token 用量
- **快捷操作**：面板顶部的编辑和执行按钮

---

## 4. 查看执行结果

在历史记录中点击某次执行可查看详细结果。

### 执行时间线

代理每一步操作的按时间排列的列表：

| 元素 | 说明 |
|------|------|
| **步骤编号** | 执行顺序 |
| **工具调用** | 调用了哪个工具（设备查询、规则创建等） |
| **参数** | 工具的输入参数 |
| **结果** | 工具执行的输出 |
| **耗时** | 该步骤的用时 |

### AI 推理过程

推理部分展示代理的思考过程：

- **思考内容**：内部推理过程（对于思考模型如 qwen3.x 或 deepseek-r1 可见）
- **决策摘要**：代理选择特定操作的原因
- **数据分析**：代理对数据的解读方式

### 性能指标

| 指标 | 说明 |
|------|------|
| **总耗时** | 从开始到完成的总时间 |
| **Token（输入）** | 发送给 LLM 的 Token 数，包括上下文和工具结果 |
| **Token（输出）** | LLM 生成的 Token 数 |
| **工具调用次数** | 执行期间调用工具的次数 |
| **状态** | 成功、失败或部分完成 |

## 5. 代理记忆

点击代理页面顶部的 **记忆** 标签页打开全局记忆存储。这是平台级别的共享知识库，所有代理都可以读取和写入。

> **注意**：记忆标签页是全局共享的，不是按代理独立的。所有代理共享同一个记忆库。

每条记忆条目包含：

| 字段 | 说明 |
|------|------|
| **类别** | 类型：设备行为、模式、偏好、事实、用户画像 |
| **内容** | 存储的知识 |
| **来源** | 创建该记忆的代理执行记录 |
| **创建时间** | 记忆被记录的时间 |
| **相关性** | 该记忆被使用的频率 |

你可以按关键词搜索记忆、按类别筛选，或按日期和相关性排序。管理记忆的操作包括：点击 **查看** 阅读完整内容、**删除** 移除单条记忆，或 **全部清除** 重置整个记忆库。

> **提示**：记忆在代理执行之间持久保存，且在服务器重启后依然存在。如果记忆变得过时（例如更换传感器后），请清除记忆，避免代理依赖过时信息。

## 6. 代理技能

点击代理页面顶部的 **技能** 标签页管理技能指南 —— 用于指导代理执行特定操作的结构化指令。技能是包含 YAML 前置信息的 Markdown 文档，定义了触发条件、分类和执行规则。

> **注意**：技能是平台级共享的 —— 任何代理都可以使用任何技能。当代理的任务或用户意图匹配到技能的触发关键词时，该技能会被注入到代理的系统提示词上下文中。

### 技能表格

技能标签页以表格形式展示所有已配置的技能：

| 列 | 说明 |
|----|------|
| **名称** | 技能名称，带有分类图标和颜色标识 |
| **类别** | 可选值：设备、规则、代理、消息、扩展、通用 |
| **优先级** | 数值 0-100，值越高在多个技能匹配时越优先 |
| **关键词** | 激活技能的触发关键词（显示前 3 个，超过则显示数量） |
| **大小** | 技能正文的字节大小 |

### 创建技能

1. 在技能标签页顶部点击 **创建技能**（或 **添加技能**）。
2. 系统打开全屏对话框，包含预加载 Markdown 模板的 **CodeMirror 编辑器**。
3. 编辑顶部的 **YAML 前置信息**：

```yaml
---
id: my-skill
name: My Skill
category: general
priority: 50
token_budget: 500
triggers:
  keywords: [example, keyword]
  tool_target:
    tool: example
    actions: [action1, action2]
anti_triggers:
  keywords: [exclude, these]
---
```

| 前置信息字段 | 必填 | 说明 |
|-------------|------|------|
| `id` | 是 | 技能的唯一标识符 |
| `name` | 是 | 技能表格中显示的名称 |
| `category` | 是 | 技能类别：`device`、`rule`、`agent`、`message`、`extension` 或 `general` |
| `priority` | 否 | 优先级 0-100（默认 50），值越高越优先 |
| `token_budget` | 否 | 技能正文的最大 Token 数（默认 500） |
| `triggers.keywords` | 是 | 激活此技能的关键词 |
| `triggers.tool_target` | 否 | 技能指导的工具和操作 |
| `anti_triggers.keywords` | 否 | 出现这些关键词时阻止技能激活 |

4. 在前置信息下方用 Markdown 编写 **技能正文**。这是技能激活时注入代理上下文的指令内容。建议包含：
   - **用途**：技能的功能说明
   - **前置条件**：执行前必须满足的条件
   - **步骤**：代理应遵循的编号指令
   - **常见错误**：已知问题和解决方案

5. 点击 **保存** 创建技能。

### 管理技能

- **查看** —— 点击眼睛图标在对话框中阅读完整的技能内容
- **编辑** —— 点击铅笔图标重新打开编辑器，技能内容已预加载
- **删除** —— 点击垃圾桶图标永久移除技能
- 技能列表分页显示（每页 10 条），支持搜索和按类别筛选

## 实用建议

### 性能优化

- **优先使用聚焦模式** —— Token 消耗更少，结果更可预测
- **设置合理的执行间隔** —— 低于 60 秒的间隔可能给 LLM 后端带来较大压力
- **只绑定必要的资源** —— 资源越少，上下文越短，Token 消耗越低
- **选择合适的模型** —— 简单监控使用快速模型（qwen3:4b），复杂推理使用强力模型（gpt-4o）
- **关注 Token 用量** —— 查看执行详情以了解成本

### 提示词示例

**异常检测**（聚焦模式，每 5 分钟执行）：

```
你是一个异常检测代理。分析绑定传感器的最新读数，
将当前值与正常范围进行对比。如果任何值超出正常范围，
生成告警，包含指标名称、当前值和预期范围。
```

**每日报告**（自由模式，cron 每天上午 9 点）：

```
你是一个日报生成代理。收集所有设备过去 24 小时的数据，
生成汇总报告，包括：
1. 每个指标的平均值、最小值和最大值
2. 触发的告警数量
3. 设备在线率
4. 值得关注的异常或趋势
以格式化的 Markdown 呈现报告。
```

**响应式告警处理**（响应式模式，事件驱动）：

```
你是一个响应式告警处理代理。当告警触发时，立即查询
相关设备分析情况。判断根本原因并在可能时采取纠正措施。
如果不宜自动操作，发送详细的通知。
```
