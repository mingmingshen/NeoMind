# Device Management

NeoMind connects to your IoT devices through protocol adapters, collects telemetry data, and enables remote control via commands. This guide walks you through the actual screens and operations.

![Devices](../../img/devices-with-data.png)

> **What you see above**: The Devices list page. Each row shows a device with its **status indicator** **(1)** (green = online, gray = offline), **name and ID** **(2)**, **type badge** **(3)**, **adapter** **(4)** (MQTT or Webhook), **transforms** **(5)**, and **last seen time** **(6)**. Click **Add Device** **(7)** to register a new device. Click any row to open the detail view.

---

## Page Overview

The Device Management page at `/devices` has three tabs:

| Tab | URL | Purpose |
|-----|-----|---------|
| **Devices** | `/devices` | All registered devices |
| **Device Types** | `/devices/types` | Templates that define metrics and commands |
| **Drafts** | `/devices/drafts` | Auto-discovered devices pending approval |

On desktop, tabs appear in the page header. On mobile, they appear as a bottom navigation bar.

---

## 1. Adding a Device

To register a new device, follow this path:

1. Navigate to **Devices** (the default tab).
2. Click **Add Device** in the top-right area of the page header.
3. The Add Device dialog opens.

![Add Device Dialog](../../img/device-add-dialog.png)

### Dialog Fields

Fill in the following fields:

**Step 1 -- Select a Device Type** (required)

Choose from the dropdown. This determines what metrics and commands the device supports. If the list is empty, create a device type first (see section 5).

**Step 2 -- Device ID** (optional)

Auto-generated by default. Click the refresh button to regenerate, or type a custom ID. This ID is used in MQTT topics and webhook URLs.

**Step 3 -- Name** (optional)

A human-readable label. If left blank, the device ID or type name is used.

**Step 4 -- Choose an Adapter**

Select **MQTT** or **Webhook** from the dropdown. The configuration section changes based on your choice.

#### MQTT Configuration

When you select MQTT, the dialog shows auto-generated topics:

| Field | Auto-Generated Value |
|-------|---------------------|
| **Telemetry Topic** | `device/{type}/{id}/uplink` |
| **Command Topic** | `device/{type}/{id}/downlink` |

The command topic field only appears if the selected device type defines commands. You can customize both topics.

Point your hardware or script to publish JSON telemetry to the telemetry topic. NeoMind's embedded MQTT broker listens on port `1883`.

#### Webhook Configuration

When you select Webhook, the dialog shows:

| Field | Value |
|-------|-------|
| **Webhook URL** | `http://{server}:9375/api/devices/{id}/webhook` |
| **Auth Token** | Click the key button to auto-generate a `whk_...` token |

Send HTTP POST requests with a JSON body to this URL. Include the token in an `Authorization: Bearer` header.

**Step 5 -- Click Create**

The device appears in the list. The status indicator shows "offline" until the device starts sending data.

## 2. Viewing the Device List

The Devices tab displays all registered devices in a responsive table.

### Table Columns

| Column | Description |
|--------|-------------|
| **Name** | Device display name and device ID in monospace below it |
| **Type** | Badge showing the device type identifier |
| **Adapter** | MQTT or Webhook, shown with an icon |
| **Transforms** | Badge showing how many data transforms are linked to this device |
| **Status** | Online (green) or Offline (gray) indicator |
| **Last Seen** | Relative timestamp of the last telemetry data received |

### Row Actions

Each row has a dropdown menu (three-dot icon on the right) with:

- **View Details** -- Opens the device detail view
- **Edit** -- Opens the edit dialog to modify name, adapter, or config
- **Delete** -- Removes the device after confirmation

On mobile, the table transforms into a card layout. Tap a card to open the detail view.

### Pagination

The list pages at 10 items per page. On mobile, infinite scroll is used instead.

## 3. Device Detail View

Click any device row (or choose "View Details" from the dropdown) to open the detail view. The URL changes to `/devices/{deviceId}`.

![Device Detail Telemetry](../../img/device-detail-telemetry.png)

### Header

The header shows:

- **Back button** -- Returns to the device list
- **Device icon and name** -- Name, type, and device ID
- **Status pill** -- Online or Offline with a pulsing dot indicator

### Device Info Card

A card near the top displays connection details:

- **Connection Method** -- The adapter type (MQTT or Webhook)
- **Device Type** -- The type identifier
- **Last Online** -- Timestamp of last received data
- **Telemetry Topic** -- For MQTT devices, the topic the device publishes to
- **Webhook URL and Token** -- For Webhook devices, the URL and masked token

### Real-Time Metrics

Below the info card, metrics are displayed in a responsive grid. Each metric card shows:

| Element | Description |
|---------|-------------|
| **Display name** | Label from the device type definition |
| **Current value** | Latest reading with appropriate formatting |
| **Unit** | Shown inline for numeric values |

Click a metric card to open a full-screen history dialog showing timestamped data points in a table, with pagination for large datasets.

**Virtual metrics** (computed by transforms) appear with a distinct purple background and a "Virtual" badge.

### Commands

If the device type defines commands, a commands section appears below the metrics. Each command is shown as a card with:

- **Command name** and display name
- **Parameter badges** showing the parameter names

Click a command card to open the send-command dialog.

## 4. Sending a Command

To send a command to a device:

1. Open the device detail view.
2. Find the command in the **Commands** section.
3. Click the command card.
4. Fill in the parameters in the dialog.
5. Click **Send Command**.

### Parameter Inputs

The dialog adapts the input control based on each parameter's data type:

| Data Type | Input Control |
|-----------|---------------|
| **String** | Text input |
| **Integer / Float** | Number input with min/max constraints |
| **Boolean** | Yes / No toggle buttons |
| **Enum** | Dropdown select with predefined options |
| **Binary** | Textarea for Base64-encoded data |
| **Array** | Textarea for JSON arrays |

After sending, a result banner shows success (green) or error (red).

> **Tip**: You can also send commands through the AI Chat. Type a natural language instruction like "Turn on the fan" and the AI will execute the command automatically.

## 5. Device Types

Device types are templates that define what metrics a device reports and what commands it supports. Every device must have a type.

Switch to the **Device Types** tab (or navigate to `/devices/types`).

![Device Types List](../../img/device-types-list.png)

### Types Table

| Column | Description |
|--------|-------------|
| **Name** | Type display name and identifier in monospace |
| **Description** | Optional description text |
| **Metrics** | Count of metric definitions (blue badge) |
| **Commands** | Count of command definitions (purple badge) |
| **Transforms** | Linked data transforms |

### Creating a Device Type

1. On the Device Types tab, click **Add Device Type**.
2. Fill in the definition form.

Two modes are available:

**Full Mode** -- Manually define each metric and command:

- **Name** and **Identifier** for the type
- **Metrics**: For each, specify name, display name, data type, unit, and optional min/max range
- **Commands**: For each, specify name, display name, and parameters with name, type, and default values

**Simple Mode** -- Paste a sample JSON payload and let the AI auto-detect the structure:

- The AI infers metric names, data types, and suggested units from your sample data
- Review and adjust the inferred definitions before saving

### Import and Export

The Device Types tab header includes additional actions:

- **Import** -- Upload a JSON file containing one or more device type definitions
- **From Cloud** -- Browse and import shared device types from the cloud library
- **Export All** -- Download all device types as a JSON file

Individual types can be exported from the row action menu.

### Example Device Type Definition

```json
{
  "device_type": "dht22_sensor",
  "name": "DHT22 Temperature & Humidity Sensor",
  "metrics": [
    { "name": "temperature", "display_name": "Temperature", "data_type": "float", "unit": "°C", "min": -40.0, "max": 80.0 },
    { "name": "humidity", "display_name": "Humidity", "data_type": "float", "unit": "%", "min": 0.0, "max": 100.0 }
  ]
}
```

## 6. Drafts (Auto-Discovery)

NeoMind can automatically detect devices that start sending MQTT data before being formally registered.

Switch to the **Drafts** tab (or navigate to `/devices/drafts`).

### How It Works

1. An unregistered device publishes JSON data to an MQTT topic matching `device/{type}/{id}/uplink`.
2. NeoMind detects the unknown device and creates a draft entry.
3. The draft appears in the Drafts tab with collected sample data.
4. You review the draft and approve or reject it.

### Draft List

Each draft row shows:

- **Status** -- Pending (yellow), Approved (green), or Rejected (red)
- **Device info** -- Inferred type and ID from the MQTT topic
- **Sample data** -- Preview of the last received payload
- **Timestamp** -- When the draft was created

### Approving a Draft

1. Click a draft row to expand its details.
2. Review the sample data and AI-inferred device type.
3. Click **Approve** to convert the draft into a full device with the suggested type.
4. The device moves to the Devices tab and begins tracking normally.

### Auto-Onboarding Configuration

Click the **Config** button on the Drafts tab to adjust settings:

| Setting | Default | Description |
|---------|---------|-------------|
| **Enabled** | On | Toggle auto-discovery on or off |
| **Max Samples** | 10 | Maximum sample payloads collected per draft |
| **Retention** | 24 hours | How long drafts are kept before automatic cleanup |

## Quick Reference

### Supported Adapters

| Adapter | Protocol | Best For | Connection |
|---------|----------|----------|------------|
| **MQTT** | MQTT v5 | Sensors, actuators, microcontrollers | Persistent TCP on port 1883 |
| **Webhook** | HTTP POST | REST-capable devices, scripts, APIs | Request/response |

### Typical Workflow

```
Create Device Type --> Add Device --> Connect Hardware --> Monitor Metrics --> Send Commands
```

1. Define a **Device Type** with metrics and commands.
2. **Add a Device** with that type and choose an adapter.
3. Configure your hardware to publish data to the generated topic or URL.
4. Monitor real-time metrics in the **Device Detail** view.
5. Send commands to control the device remotely.
