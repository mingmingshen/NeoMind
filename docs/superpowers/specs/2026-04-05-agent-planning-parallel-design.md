# Agent Planning & Parallel Execution Design

**Date:** 2026-04-05
**Status:** Draft
**Scope:** `crates/neomind-agent/` + `web/src/`

## Problem

The current agent pipeline relies on the LLM to organically produce tool calls, often one at a time per response round. While `streaming.rs` already executes multiple tool calls from a single LLM round in parallel (via `futures::join_all`), the LLM frequently returns only one tool call per round, requiring multiple round-trips for multi-device queries. There is no explicit planning step that deliberately constructs multi-tool-call plans to exploit this existing parallelism.

Additionally, users have no visibility into what the agent plans to do before it does it. The existing `AgentEvent::Plan` event only carries a single `(step, stage)` string — there is no structured execution plan with dependency tracking.

## Goals

1. Add lightweight planning before tool execution — deliberately construct multi-tool-call plans to exploit existing parallel execution
2. Users see what the agent plans to do before execution begins
3. Use a hybrid approach: keyword-based fast path for simple tasks, LLM-based planning for complex ones
4. Frontend shows structured execution plan with real-time step progress

## Non-Goals

- Multi-agent orchestration or inter-agent communication
- Self-reflection or evaluator-optimizer loops (future work)
- Replacing the existing parallel execution in `streaming.rs` — the planner feeds into it

## Architecture

### Core Types

```rust
/// Step identifier — simple usize index into the plan's step vector
type StepId = usize;

/// Execution plan generated by the planner
struct ExecutionPlan {
    steps: Vec<PlanStep>,
    planning_mode: PlanningMode,
}

struct PlanStep {
    id: StepId,
    tool_name: String,       // "device", "agent", "rule", etc.
    action: String,          // "list", "get", "query", "control", etc.
    params: serde_json::Value,
    depends_on: Vec<StepId>, // empty = parallelizable within the same batch
}

#[derive(Debug, Clone, Serialize, Deserialize)]
enum PlanningMode {
    Keyword,   // Fast path, rule-based
    LLM,       // Deep path, LLM-generated
}
```

Note: `estimated_complexity` field removed — YAGNI. The planner doesn't need it.

### Pipeline Change

The planning stage runs **before** entering the streaming loop in `streaming.rs`. It is a pre-processing step that produces a structured plan, which the streaming loop then executes using the existing `join_all` parallel pattern.

```
Current flow:
  streaming_loop { LLM call → parse tool calls → join_all execute → loop }

New flow:
  Plan = Planner::plan(intent, context)
  ↓
  streaming_loop { feed plan's tool calls to LLM → LLM responds with planned calls → join_all execute → loop }
```

The planner does **not** replace the streaming loop. It influences what tool calls the LLM is guided to produce by providing context about the planned steps. This means:

1. **Keyword path**: The planner constructs tool calls directly and injects them as "suggested" tool calls into the conversation context, so the LLM can validate and execute them.
2. **LLM path**: The planner uses one lightweight LLM call to produce a structured plan, then feeds it into the normal streaming loop.

### AgentEvent Changes

The existing `AgentEvent::Plan { step, stage }` event remains for backward compatibility. New events are additive:

```rust
// New variants added to AgentEvent enum
ExecutionPlanCreated {
    plan: ExecutionPlan,        // Full plan for frontend rendering
}
PlanStepStarted {
    step_id: StepId,
    description: String,        // Human-readable step description
}
PlanStepCompleted {
    step_id: StepId,
    success: bool,
    summary: String,            // Brief result summary
}
```

Relationship to existing events:
- `AgentEvent::Plan { step, stage }` → kept, still emitted for simple single-step flows
- `AgentEvent::ToolCallStart` / `ToolCallEnd` → still emitted for each tool call; `PlanStepStarted`/`Completed` are higher-level wrappers
- `ExecutionPlanCreated` replaces `Plan` for multi-step scenarios; frontend should prefer the new events when present

The `ParallelExecutionStarted`/`ParallelExecutionCompleted` events from the initial draft are removed — they are redundant with the existing `ToolCallStart`/`ToolCallEnd` pair plus `join_all` in `streaming.rs`. Frontend can infer parallelism from concurrent `PlanStepStarted` events.

## Component Design

### KeywordPlanner (Fast Path)

Located in `crates/neomind-agent/src/agent/planner/keyword.rs`.

Maps `IntentCategory` (from `staged.rs`) to execution templates:

| IntentCategory | Sub-intent Detection | Strategy |
|---|---|---|
| `Device` | Parse action: query vs control (from context entities) | Single query → 1 step; Multi-query → N parallel steps; Control → confirm + control (serial) |
| `Rule` | Default | Single step (list/get/delete) |
| `Data` | Default | Query step → optional analysis step (serial) |
| `Alert` | Default | Single step |
| `Workflow` | Multi-domain | Falls through to LLM planner |
| `System` | Default | Skip planning, use existing flow |
| `Help` | Default | Skip planning |
| `General` | Default | Skip planning |

Sub-intent detection (query vs control within `Device` category) is done by analyzing the `ContextBundle` entities and the user message keywords, not by a separate classification pass.

Key behaviors:
- Zero additional LLM calls
- Parallel detection: multiple independent device operations auto-parallelized
- Falls through to LLM planner when intent is ambiguous

### LLMPlanner (Deep Path)

Located in `crates/neomind-agent/src/agent/planner/llm_planner.rs`.

One lightweight LLM call that outputs a structured JSON execution plan.

Prompt design (follows Anthropic ACI best practices):
- Provide available aggregated tool schemas (reuse from `toolkit/aggregated.rs`)
- Request JSON output: `{ "steps": [{ "tool": "device", "action": "query", "params": {...}, "depends_on": [] }] }`
- Explicitly instruct to mark independent steps with empty `depends_on`
- Tool names use actual names: `"device"`, `"agent"`, `"agent_history"`, `"rule"`, `"alert"`, `"extension"`
- Actions use actual action names: `"list"`, `"get"`, `"query"`, `"control"`, etc.

Fallback: If LLM planning fails or times out (2s budget), fall back to KeywordPlanner output.

### Integration with Existing Streaming Loop

The planner runs as a **pre-processing step** before the main streaming loop in `streaming.rs`:

```
1. IntentClassifier classifies intent (existing Stage 1 in staged.rs)
2. Planner produces ExecutionPlan (new, before streaming)
3. ExecutionPlan is converted to tool call suggestions
4. Normal streaming loop runs, using plan context to guide LLM tool calls
5. Existing join_all in streaming.rs handles actual parallel execution
6. New plan events are emitted alongside existing ToolCallStart/End events
```

Both `process_stream_events` and `process_multimodal_stream_events` paths receive the optional `ExecutionPlan` and emit plan events when present.

### Path Selection Logic

```
IntentClassifier output
  │
  ├─ confidence >= 0.8
  │   AND single IntentCategory (not Workflow)
  │   AND identified entities <= 3
  │   → KeywordPlanner (fast path)
  │
  └─ confidence < 0.8
      OR IntentCategory::Workflow
      OR identified entities > 3
      OR user message contains multi-step reasoning cues
      → LLMPlanner (deep path)
```

## Frontend Design

### ExecutionPlanPanel Component

A collapsible panel that appears in the chat message stream:

```
┌─ Execution Plan ─────────────────────────┐
│ Query living_room temp sensor       ✅   │  ← Parallel Group 1
│ Query bedroom temp sensor           ✅   │
│ Analyze temperature trend           ⏳   │  ← Waits for group 1
├──────────────────────────────────────────┤
│ living_room: 24.5°C                     │  ← Real-time results
│ bedroom: 22.1°C                         │
│ Analyzing...                            │
└──────────────────────────────────────────┘
```

State management:
- Plan state lives in `sessionSlice` (not a new slice) under an `activePlan` field
- `ChatContainer.tsx` useReducer handles new WebSocket events and syncs to store
- Plan panel auto-collapses after all steps complete

Backward compatibility: Old frontend versions ignore unknown `ServerMessage` discriminants in the `switch` statement — no breakage.

## Rollout Strategy

- `AgentConfig` gets a `planning: PlanningConfig` field with `enabled: bool` defaulting to `true`
- When `planning.enabled = false`, the entire planning stage is skipped and the existing flow runs unchanged
- This allows instant rollback if issues arise
- The `PlanningConfig` can be set per-session or globally via the API

## Error Handling

| Scenario | Behavior |
|---|---|
| LLM Planner fails/times out | Fall back to KeywordPlanner |
| Keyword Planner produces empty plan | Skip planning, use existing sequential flow |
| Plan step fails during execution | Mark step as failed in event, continue remaining steps |
| All steps fail | Return error message, no retry |
| Plan exceeds max steps (10) | Truncate and warn |

## Safety Constraints for Destructive Operations

Destructive operations are identified by the `(tool_name, action)` pair:

| Tool | Action | Classification |
|---|---|---|
| `device` | `control` | Destructive — always serial, requires confirmation |
| `device` | `list`/`get`/`query` | Safe — parallelizable |
| `rule` | `delete` | Destructive — serial |
| `rule` | `list`/`get` | Safe — parallelizable |
| `agent` | `create`/`update`/`control` | Destructive — serial |
| `alert` | `create`/`acknowledge` | Destructive — serial |
| All others | — | Safe — parallelizable |

## Testing Strategy

1. **Unit tests**: KeywordPlanner template mapping with actual `IntentCategory` values, LLMPlanner JSON output parsing, plan-to-tool-call conversion
2. **Integration tests**: End-to-end planning → streaming loop → response with mock tools
3. **Performance benchmarks**: Compare multi-round (current) vs planned (new) for 3/5/10 device queries
4. **Fallback tests**: Verify graceful degradation when LLM planning fails
5. **Backward compat tests**: Verify existing single-tool-call flows are unchanged when `planning.enabled = false`

## Success Metrics

- Multi-device query latency reduced by 40-60% (fewer LLM round-trips via planned multi-tool calls)
- LLM planner fallback rate < 5% for common scenarios
- No regression in single-tool-call scenarios (backend processing overhead < 50ms for keyword path, not including network/React rendering)
- Frontend plan panel renders within 100ms of receiving events
