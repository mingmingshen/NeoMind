# Agent Planning & Parallel Execution Design

**Date:** 2026-04-05
**Status:** Draft
**Scope:** `crates/neomind-agent/` + `web/src/`

## Problem

The current agent uses a 4-stage pipeline (Intent → Tool Selection → Tool Execution → Response) where tools execute sequentially. For multi-device queries and complex tasks this creates unnecessary latency and token waste. The agent also lacks explicit planning, making it harder for users to understand what the agent will do before it does it.

## Goals

1. Add lightweight planning before tool execution — users see what the agent plans to do
2. Execute independent tool calls in parallel — reduce latency for multi-device scenarios
3. Use a hybrid approach: keyword-based fast path for simple tasks, LLM-based planning for complex ones
4. Frontend shows execution plan with real-time step progress

## Non-Goals

- Multi-agent orchestration or inter-agent communication
- Self-reflection or evaluator-optimizer loops (future work)
- Changing the existing `AgentEvent` enum signature (additive only)

## Architecture

### Execution Plan

```rust
/// Execution plan generated by the planner
struct ExecutionPlan {
    steps: Vec<PlanStep>,
    planning_mode: PlanningMode,
}

struct PlanStep {
    id: StepId,
    tool_name: String,
    action: String,
    params: Value,
    depends_on: Vec<StepId>,       // empty = parallelizable
    estimated_complexity: Complexity,
}

enum PlanningMode {
    Keyword,   // Fast path, rule-based
    LLM,       // Deep path, LLM-generated
}

enum Complexity {
    Simple,
    Medium,
    Complex,
}
```

### Pipeline Change

Current 4-stage pipeline becomes 5-stage in `staged.rs`:

```
Stage 1: Intent Classification     (existing, unchanged)
Stage 2: Planning                  (NEW)
  ├─ KeywordPlanner (if confidence >= 0.8 && single-domain intent)
  └─ LLMPlanner     (if confidence < 0.8 || multi-domain intent)
Stage 3: Tool Execution            (ENHANCED - supports parallel)
Stage 4: Response Generation       (existing, unchanged)
```

### New AgentEvent Variants

Additive only — existing consumers unaffected:

```rust
// Added to AgentEvent enum
PlanGenerated { plan: ExecutionPlan }
PlanStepStarted { step_id: StepId }
PlanStepCompleted { step_id: StepId, result: Value }
ParallelExecutionStarted { step_ids: Vec<StepId> }
ParallelExecutionCompleted { results: Vec<(StepId, Value)> }
```

## Component Design

### KeywordPlanner (Fast Path)

Located in `crates/neomind-agent/src/agent/planner/keyword.rs`.

Maps intent classification results to predefined execution templates:

| Intent Type | Strategy |
|---|---|
| `DeviceQuery` (single) | 1 step, no planning overhead |
| `DeviceQuery` (multi) | N parallel steps |
| `DeviceControl` | Confirm step → control step (serial) |
| `DataAnalysis` | Query step → analyze step (serial) |
| `RuleManagement` | Single step |
| `Alert` | Single step |
| `General` | Skip planning, use existing flow |

Key behaviors:
- Zero additional LLM calls
- Parallel detection: multiple independent device operations auto-parallelized
- Falls through to LLM planner when intent is ambiguous

### LLMPlanner (Deep Path)

Located in `crates/neomind-agent/src/agent/planner/llm_planner.rs`.

One lightweight LLM call that outputs a structured execution plan:

Prompt design:
- Provide available tool list with descriptions (reuse aggregated tool schemas)
- Request JSON output with `steps` array and `depends_on` fields
- Explicitly instruct to mark independent steps as parallelizable
- Use a fast/small model endpoint when available

Fallback: If LLM planning fails or times out (2s budget), fall back to KeywordPlanner output.

### ParallelExecutor

Located in `crates/neomind-agent/src/agent/parallel_executor.rs`.

```rust
struct ParallelExecutor {
    tool_registry: Arc<ToolRegistry>,
    max_parallelism: usize,  // default: 4
}
```

Execution logic:
1. Topological sort of plan steps by dependency
2. Group independent steps into parallel batches
3. Execute each batch with `tokio::join!` / `futures::join_all`
4. Emit `ParallelExecutionStarted` / `ParallelExecutionCompleted` events
5. Wait for dependent steps before executing next batch

Safety constraints:
- Max parallelism: 4 (configurable via `AgentConfig`)
- Destructive operations (`device:control`) default to serial unless explicitly safe
- Total timeout inherits existing `StreamSafeguards.max_stream_duration`
- Per-step timeout: 30s

### Path Selection Logic

```
IntentClassifier output
  │
  ├─ confidence >= 0.8
  │   AND single-domain intent (e.g., only devices, only rules)
  │   AND identified entities <= 3
  │   → KeywordPlanner (fast path)
  │
  └─ confidence < 0.8
      OR multi-domain intent
      OR identified entities > 3
      OR user message contains multi-step reasoning cues
      → LLMPlanner (deep path)
```

## Frontend Design

### ExecutionPlanPanel Component

A collapsible panel that appears in the chat message stream:

```
┌─ Execution Plan ─────────────────────────┐
│ 🔍 Query living_room temp sensor   ✅    │  ← Parallel Group 1
│ 🔍 Query bedroom temp sensor       ✅    │
│ 📊 Analyze temperature trend        ⏳   │  ← Waits for group 1
├──────────────────────────────────────────┤
│ ✅ living_room: 24.5°C                  │  ← Real-time results
│ ✅ bedroom: 22.1°C                      │
│ ⏳ Analyzing...                          │
└──────────────────────────────────────────┘
```

State management:
- Zustand `executionPlan` slice stores active plans per session
- WebSocket handler maps new events to plan state updates
- Plan panel auto-collapses after all steps complete

Backward compatibility: Old frontend versions ignore unknown event types — no breakage.

## Error Handling

| Scenario | Behavior |
|---|---|
| LLM Planner fails/times out | Fall back to KeywordPlanner |
| Keyword Planner produces empty plan | Skip planning, use existing sequential flow |
| Parallel step fails | Mark step as failed, continue remaining steps |
| All steps fail | Return error message, no retry |
| Plan exceeds max steps (10) | Truncate and warn |

## Testing Strategy

1. **Unit tests**: KeywordPlanner template mapping, LLMPlanner output parsing, ParallelExecutor topology
2. **Integration tests**: End-to-end planning → execution → response with mock tools
3. **Performance benchmarks**: Compare parallel vs sequential for 3/5/10 device queries
4. **Fallback tests**: Verify graceful degradation when LLM planning fails

## Success Metrics

- Multi-device query latency reduced by 40-60% (parallel execution)
- LLM planner fallback rate < 5% for common scenarios
- No regression in single-tool-call scenarios (overhead < 50ms for keyword path)
- Frontend plan panel renders within 100ms of receiving events
