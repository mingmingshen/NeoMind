# Embedded MQTT Broker Authentication & TLS Management

**Date**: 2026-05-27
**Status**: Draft
**Scope**: Built-in MQTT broker security (auth, TLS) with API/UI management

## Background

NeoMind's embedded MQTT broker (rumqttd 0.20) currently runs without authentication or TLS. Any device on the network can connect to `0.0.0.0:1883` with no credentials. This is acceptable for development but a security risk in production.

rumqttd already supports `auth: HashMap<String, String>` (static credentials) and `tls: TlsConfig` (Rustls-based TLS). We just pass `None` / `None` today.

## Requirements

1. **Username/password authentication** — configurable per-broker, optional
2. **TLS support** — server certificate, optional client CA verification
3. **API & UI management** — CRUD for credentials, TLS cert upload, broker config
4. **Internal bypass** — NeoMind's own MQTT adapter must never be blocked by auth
5. **Broker restart** — config changes trigger a fast broker restart (< 100ms)

## Architecture

### Internal Bypass Pattern

When auth is enabled, a hidden system credential is injected automatically:

```
NeoMind internal adapter → 127.0.0.1:1883
  user: "__neomind_internal__"
  pass: <auto-generated, stored in redb>

External devices → 0.0.0.0:1883
  user: <user-configured>
  pass: <user-configured>
```

- System credential is invisible in UI, cannot be deleted by users
- Internal adapter code reads the system credential at startup and reconnects after broker restart
- External devices must provide valid credentials when auth is enabled

### Broker Restart Strategy

rumqttd loads config at startup and does not support hot-reload. The restart flow:

```
API request → validate config → save to redb
  → send stop signal to old broker thread
  → join old thread (blocks until stopped)
  → create new Broker with updated config
  → start new broker thread
  → internal adapter reconnects automatically
```

Target restart window: < 100ms. MQTT clients (devices) have built-in reconnect logic and will re-establish connections automatically.

## Data Model

### EmbeddedBrokerConfig (extended)

```rust
// crates/neomind-devices/src/embedded_broker.rs

pub struct EmbeddedBrokerConfig {
    // Existing fields
    pub listen: String,              // default: "0.0.0.0"
    pub port: u16,                   // default: 1883
    pub max_connections: usize,      // default: 1000
    pub max_payload_size: usize,     // default: 256MB
    pub connection_timeout_ms: u16,  // default: 60000
    pub dynamic_filters: bool,       // default: true

    // New: Authentication
    pub auth_enabled: bool,          // default: false
    pub credentials: Vec<MqttCredential>,

    // New: TLS
    pub tls_enabled: bool,           // default: false
    pub tls_cert_path: Option<String>,   // Server certificate file path
    pub tls_key_path: Option<String>,    // Server private key file path
    pub tls_ca_path: Option<String>,     // CA certificate (optional, for client verification)
}

pub struct MqttCredential {
    pub username: String,
    pub password_hash: String,  // bcrypt hash
}
```

### Storage

- Config stored in `settings.redb` under table `embedded_broker_config`
- TLS certificates stored as files in `data/tls/` directory:
  - `data/tls/server.cert.pem` — server certificate
  - `data/tls/server.key.pem` — server private key
  - `data/tls/ca.cert.pem` — CA certificate (optional)
- System internal credential stored alongside user credentials, flagged with `__neomind_internal__` username prefix

### Password Security

- Passwords hashed with bcrypt before storage
- API responses always mask passwords (show `••••••••`)
- Raw passwords only accepted in create/update requests, never returned

## API Endpoints

### Get Broker Config

```
GET /api/mqtt/broker-config
```

Response:
```json
{
  "success": true,
  "data": {
    "listen": "0.0.0.0",
    "port": 1883,
    "max_connections": 1000,
    "auth_enabled": false,
    "credentials": [
      { "username": "device-user", "password": "••••••••" }
    ],
    "tls_enabled": false,
    "tls_cert_path": null,
    "tls_key_path": null,
    "tls_ca_path": null
  }
}
```

System credential (`__neomind_internal__`) is excluded from response.

### Update Broker Config

```
PUT /api/mqtt/broker-config
```

Request body:
```json
{
  "listen": "0.0.0.0",
  "port": 1883,
  "auth_enabled": true,
  "tls_enabled": false
}
```

Flow:
1. Validate config (port range, TLS cert files exist if enabled)
2. Save to `settings.redb`
3. Restart broker with new config
4. Return updated config

### Add Credential

```
POST /api/mqtt/broker-config/credentials
```

Request body:
```json
{
  "username": "device-user",
  "password": "my-secret-password"
}
```

Flow:
1. Validate username (not empty, not `__neomind_internal__` prefix)
2. Hash password with bcrypt
3. Save to credentials list in redb
4. If broker is running with auth enabled, restart broker

### Delete Credential

```
DELETE /api/mqtt/broker-config/credentials/:username
```

Flow:
1. Check username is not system credential
2. Remove from credentials list
3. If broker is running with auth enabled, restart broker

### Upload TLS Certificate

```
PUT /api/mqtt/broker-config/tls
```

Request body (multipart or JSON):
```json
{
  "cert_pem": "-----BEGIN CERTIFICATE-----\n...",
  "key_pem": "-----BEGIN PRIVATE KEY-----\n...",
  "ca_pem": "-----BEGIN CERTIFICATE-----\n..."  // optional
}
```

Flow:
1. Validate PEM format
2. Write files to `data/tls/` directory
3. Update config paths
4. If TLS is enabled, restart broker

## Backend Changes

### 1. `embedded_broker.rs` — Core Changes

- Expand `EmbeddedBrokerConfig` with auth/TLS fields
- Add `stop: Arc<Notify>` signal for graceful shutdown
- `start()` method: build rumqttd config with `auth` HashMap and `tls` TlsConfig
- `restart()` method: stop old broker, start new one
- System credential auto-injection: when `auth_enabled`, always include `__neomind_internal__` in the auth HashMap with an auto-generated password

### 2. `settings.rs` — Storage

- New table `embedded_broker_config` in settings redb
- `get_embedded_broker_config()` — load from redb, fall back to defaults
- `save_embedded_broker_config()` — save full config
- `get_system_mqtt_credential()` — read the internal system credential (used by MQTT adapter)

### 3. MQTT Adapter — Internal Connection

- When connecting to embedded broker, check if auth is enabled
- If enabled, fetch system credential from settings and use it
- On connection loss (broker restart), reconnect with same credential

### 4. API Handlers — `mqtt/brokers.rs`

- New handler functions for the endpoints above
- Config validation logic
- Broker restart orchestration

### 5. Server Init — `server/types.rs`

- `init_device_adapters()`: load broker config from redb (not just config.toml)
- Pass system credential to internal MQTT adapter
