# Embedded MQTT Broker Authentication & TLS Management

**Date**: 2026-05-27
**Status**: Draft
**Scope**: Built-in MQTT broker security (auth, TLS) with API/UI management

## Background

NeoMind's embedded MQTT broker (rumqttd 0.20) currently runs without authentication or TLS. Any device on the network can connect to `0.0.0.0:1883` with no credentials. This is acceptable for development but a security risk in production.

rumqttd supports two authentication mechanisms:
- `auth: Option<HashMap<String, String>>` — static plaintext username/password pairs, loaded at startup
- `external_auth: Option<AuthHandler>` — async callback `(ClientId, AuthUser, AuthPass) -> bool`, invoked on every connection

**Decision: use `external_auth`**. Reasons:
- Allows bcrypt-hashed password storage (static `auth` requires plaintext comparison via `ct_eq`)
- Credentials can be changed at runtime without restarting the broker
- The callback reads from redb on each connection, so changes are immediately effective
- Only TLS/port/listen-address changes require broker restart

## Requirements

1. **Username/password authentication** — optional, configurable via API/UI
2. **TLS support** — server certificate, optional client CA verification
3. **API & UI management** — CRUD for credentials, TLS cert upload, broker config
4. **Internal bypass** — NeoMind's own MQTT adapter must never be blocked by auth
5. **Hot credential reload** — credential changes take effect immediately, no broker restart needed
6. **Broker restart** — only required for TLS, port, or listen-address changes

## Architecture

### Authentication Flow (external_auth)

```
Device connects with username/password
  → rumqttd calls external_auth(client_id, username, password)
  → callback reads credentials from redb (Arc<redb::Database>)
  → checks system credential first (username == "__neomind_internal__")
  → then checks user credentials (bcrypt::verify)
  → returns true/false
```

The `external_auth` callback is set once at broker startup and holds an `Arc<redb::Database>` reference. It reads credentials fresh from redb on every connection, so adding/removing credentials via API takes effect immediately.

### Internal Bypass Pattern

When auth is enabled, a hidden system credential is auto-generated:

```
NeoMind internal adapter → 127.0.0.1:1883
  user: "__neomind_internal__"
  pass: <auto-generated, stored in separate redb key>

External devices → 0.0.0.0:1883
  user: <user-configured>
  pass: <user-configured>
```

- System credential stored in a **separate redb key** (`system::mqtt_internal_credential`), not in the user credentials table
- Auto-generated at first startup when auth is enabled, using a cryptographically random password
- Internal adapter reads system credential at startup and uses it when connecting
- On broker restart, internal adapter reconnects with same credential
- System credential is invisible in UI and cannot be deleted/modified by users

### Broker Restart Strategy

Only needed for: TLS enable/disable, port change, listen-address change.

**NOT needed for**: credential add/delete, auth enable/disable (handled by `external_auth` callback dynamically).

When restart is required:

```
API request → validate config → save to redb
  → set stop flag (AtomicBool)
  → old broker thread detects stop flag, breaks accept loop
  → join old thread
  → create new Broker with updated config
  → start new broker thread
  → internal adapter reconnects automatically
```

**Stop mechanism**: The broker thread runs `rumqttd::Broker::start()` which internally runs an accept loop. To stop it:
1. Set `stop: Arc<AtomicBool>` flag
2. Open a TCP connection to the broker's own port to unblock the accept call
3. The broker thread checks the stop flag after each accept cycle
4. When flagged, the thread exits gracefully

This is the pragmatic approach since rumqttd doesn't expose a shutdown API. The TCP self-connection trick unblocks the accept loop so the stop flag can be checked.

## Data Model

### EmbeddedBrokerConfig (extended)

```rust
// crates/neomind-devices/src/embedded_broker.rs

pub struct EmbeddedBrokerConfig {
    // Existing fields
    pub listen: String,              // default: "0.0.0.0"
    pub port: u16,                   // default: 1883
    pub max_connections: usize,      // default: 1000
    pub max_payload_size: usize,     // default: 256MB
    pub connection_timeout_ms: u16,  // default: 60000
    pub dynamic_filters: bool,       // default: true

    // New: Authentication
    pub auth_enabled: bool,          // default: false

    // New: TLS
    pub tls_enabled: bool,           // default: false
    pub tls_cert_path: Option<String>,   // Server certificate file path
    pub tls_key_path: Option<String>,    // Server private key file path
    pub tls_ca_path: Option<String>,     // CA certificate (optional, for client verification)
}

// Stored separately in user credentials table
pub struct MqttCredential {
    pub username: String,
    pub password_hash: String,  // bcrypt hash
}
```

Note: `credentials` list is NOT in `EmbeddedBrokerConfig`. It's stored as a separate redb table, read dynamically by the `external_auth` callback. The config only has the `auth_enabled` flag.

### Storage

- **Broker config** (`embedded_broker_config` table in `settings.redb`): listen, port, auth_enabled, tls_enabled, tls paths
- **User credentials** (`mqtt_credentials` table in `settings.redb`): username → bcrypt hash pairs
- **System credential** (`system::mqtt_internal_credential` key in `settings.redb`): auto-generated password for internal adapter (plaintext, since it's passed to rumqttc directly)
- **TLS certificates** as files in `data/tls/` directory:
  - `data/tls/server.cert.pem` — server certificate
  - `data/tls/server.key.pem` — server private key
  - `data/tls/ca.cert.pem` — CA certificate (optional)

### Password Security

- User passwords hashed with bcrypt before storage in redb
- `external_auth` callback uses `bcrypt::verify()` to check incoming passwords
- System credential stored as plaintext in a separate key (never exposed via API)
- API responses always mask passwords (show `••••••••`)
- Raw passwords only accepted in create/update requests, never returned
- Credential count limit: 100 (prevents slow startup from large auth HashMap)

## API Endpoints

### Get Broker Config

```
GET /api/mqtt/broker-config
```

Response:
```json
{
  "success": true,
  "data": {
    "listen": "0.0.0.0",
    "port": 1883,
    "max_connections": 1000,
    "auth_enabled": false,
    "credentials": [
      { "username": "device-user", "password": "••••••••" }
    ],
    "tls_enabled": false,
    "tls_cert_path": null,
    "tls_key_path": null,
    "tls_ca_path": null
  }
}
```

System credential is excluded from response. Only user-managed credentials are shown.

### Update Broker Config

```
PUT /api/mqtt/broker-config
```

Request body:
```json
{
  "listen": "0.0.0.0",
  "port": 1883,
  "auth_enabled": true,
  "tls_enabled": false
}
```

Flow:
1. Validate config (port range, TLS cert files exist if tls_enabled=true)
2. Save to `settings.redb`
3. Check if restart is needed (port/listen/tls changes)
4. If restart needed: stop old broker → start new broker
5. If only auth_enabled changed: no restart (external_auth handles it dynamically)
6. Return updated config

### Add Credential

```
POST /api/mqtt/broker-config/credentials
```

Request body:
```json
{
  "username": "device-user",
  "password": "my-secret-password"
}
```

Flow:
1. Validate username (3-64 chars, alphanumeric + `_` + `-`, not `__neomind_internal__`)
2. Check credential count ≤ 100
3. Hash password with bcrypt
4. Save to `mqtt_credentials` table in redb
5. **No broker restart needed** — external_auth reads redb dynamically

### Delete Credential

```
POST /api/mqtt/broker-config/credentials/delete
```

Request body:
```json
{
  "username": "device-user"
}
```

Note: Uses POST with JSON body instead of DELETE with path param to avoid URL encoding issues with special characters in usernames.

Flow:
1. Check username is not system credential
2. Remove from `mqtt_credentials` table
3. **No broker restart needed**

### Upload TLS Certificate

```
PUT /api/mqtt/broker-config/tls
```

Request body:
```json
{
  "cert_pem": "-----BEGIN CERTIFICATE-----\n...",
  "key_pem": "-----BEGIN PRIVATE KEY-----\n...",
  "ca_pem": "-----BEGIN CERTIFICATE-----\n..."
}
```

Flow:
1. Validate PEM format (parse certificates, check key matches cert)
2. Write files to `data/tls/` directory
3. Update config paths in redb
4. If TLS is enabled, restart broker

## Backend Changes

### 1. `embedded_broker.rs` — Core Changes

**Struct redesign**: `EmbeddedBroker` must hold lifecycle state for restart capability:

```rust
pub struct EmbeddedBroker {
    config: EmbeddedBrokerConfig,
    running: Arc<AtomicBool>,
    stop: Arc<AtomicBool>,           // NEW: signal to stop broker thread
    thread_handle: Mutex<Option<JoinHandle<()>>>,  // NEW: hold thread handle
    db: Arc<Database>,               // NEW: redb handle for external_auth
}
```

**Key methods**:
- `start()`: builds rumqttd config, sets `external_auth` callback with `Arc<Database>`, spawns thread
- `stop()`: sets stop flag, sends TCP self-connection to unblock accept, joins thread
- `restart()`: stop() then start()
- `is_running()`: checks AtomicBool

**external_auth callback setup**:
```rust
let db = self.db.clone();
let system_cred_username = "__neomind_internal__".to_string();

server_settings.set_auth_handler(move |_client_id, username, password| {
    let db = db.clone();
    async move {
        // Check system credential first
        if username == system_cred_username {
            let stored = read_system_credential(&db);
            return stored.map(|p| p == password).unwrap_or(false);
        }
        // Check user credentials
        let hash = read_credential_hash(&db, &username);
        hash.map(|h| bcrypt::verify(&password, &h).unwrap_or(false)).unwrap_or(false)
    }
});
```

**TLS config**: When `tls_enabled`, set `ServerSettings.tls` to `TlsConfig::Rustls { certpath, keypath, ca }`.

### 2. `settings.rs` — Storage

- New table `embedded_broker_config` in settings redb
- New table `mqtt_credentials` in settings redb
- `get_embedded_broker_config()` — load from redb, fall back to defaults
- `save_embedded_broker_config()` — save config
- `get_mqtt_credentials()` — list all user credentials
- `add_mqtt_credential(username, hash)` — add credential
- `delete_mqtt_credential(username)` — remove credential
- `get_system_mqtt_credential()` — read system internal credential
- `ensure_system_credential()` — auto-generate if not exists

### 3. MQTT Adapter — Internal Connection

- When connecting to embedded broker, check if auth is enabled via `get_embedded_broker_config()`
- If enabled, fetch system credential via `get_system_mqtt_credential()` and pass as username/password
- On connection loss (broker restart), reconnect with same credential

### 4. API Handlers — `mqtt/brokers.rs`

- New handler functions for the endpoints above
- Config validation logic (port range, PEM validation, file existence)
- Restart orchestration with error handling
- Rate limiting: max 5 restart-triggering operations per minute (config changes, TLS uploads)

### 5. Server Init — `server/types.rs`

- `init_device_adapters()`: load broker config from redb instead of just config.toml
- Pass `Arc<Database>` to `EmbeddedBroker::new()` for external_auth
- Pass system credential to internal MQTT adapter

## Edge Cases & Error Handling

1. **Broker fails to start on new port**: Roll back to previous config, keep broker running on old port
2. **TLS cert invalid**: Validate before saving, reject invalid PEM or key/cert mismatch
3. **All credentials deleted with auth enabled**: Broker still runs but no external device can connect (by design)
4. **System credential corrupted**: Auto-regenerate on next broker start, log a warning
5. **Port race on restart**: Use SO_REUSEADDR + poll for port availability with 200ms timeout
6. **Concurrent config updates**: Mutex on broker restart operation, return 409 Conflict if restart in progress
7. **Credential add during broker restart**: Safe — redb writes succeed, external_auth reads fresh on next connection
8. **In-flight messages during restart**: QoS 0 messages may be lost. QoS 1/2 survive via MQTT protocol. UI warns user before restart.
