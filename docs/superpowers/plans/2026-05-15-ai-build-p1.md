# AI Build Mode - Phase 1: Core Infrastructure

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build the AI Build floating panel UI and `system` aggregated tool, enabling users to open a global AI assistant that can execute native machine operations.

**Architecture:** A floating AI panel (AIBuildFab + AIBuildPanel) connects to the existing Agent WebSocket session system. A new `system` aggregated tool follows the established `AggregatedToolsBuilder` pattern with actions for file I/O, shell execution, network scanning, and hardware queries. The tool wraps the existing `ShellTool` for command execution with an additional security validation layer.

**Tech Stack:** Rust (Axum, neomind-agent toolkit), React 18 + TypeScript + Zustand + Tailwind, WebSocket

**Spec:** `docs/superpowers/specs/2026-05-15-ai-build-mode-design.md` (Sections 2, 4, 12, 14)

## Task 1: SystemTool — File Operations

**Files:**
- Create: `crates/neomind-agent/src/toolkit/system.rs`
- Modify: `crates/neomind-agent/src/toolkit/mod.rs`

- [ ] **Step 1: Create system.rs with file operation actions**

Create `crates/neomind-agent/src/toolkit/system.rs` implementing the `Tool` trait following the `DeviceTool` pattern in `aggregated.rs`. The tool name is `"system"`. Initial actions: `fs_read_file`, `fs_write_file`, `fs_list_dir`, `fs_exists`.

```rust
// Key structure:
pub struct SystemTool {
    data_dir: PathBuf,
    shell_config: ShellConfig,
}

// Tool trait:
// name() → "system"
// description() → docstring listing all actions with params
// parameters() → JSON schema with "action" enum + action-specific params
// category() → ToolCategory::System (new variant or reuse)
// execute() → match action, route to handler
```

Action handlers:
- `fs_read_file` → `std::fs::read_to_string`, return content (reject if file > 1MB)
- `fs_write_file` → requires confirmation via ConfirmActionTool pattern, write content to path
- `fs_list_dir` → `std::fs::read_dir`, return file/dir names
- `fs_exists` → `std::path::Path::exists`, return bool

Security: `fs_write_file` validates path doesn't escape allowed dirs (reject `..` traversal above data_dir). `fs_read_file` blocks `/etc/shadow`, `.env`, credential files.


## Task 2: SystemTool — Shell + Network + Hardware Actions

**Files:**
- Modify: `crates/neomind-agent/src/toolkit/system.rs`

- [ ] **Step 1: Add shell execution actions**

Add actions to `SystemTool`:
- `exec` → Execute arbitrary shell command via `std::process::Command` (same approach as ShellTool), with whitelist/blacklist validation before execution
- `exec_readonly` → Same but restricted to read-only commands (ls, cat, head, tail, find, grep, ip, ifconfig, df, free, uname, hostname, whoami, pwd, echo, stat, file, wc, du, ps, top -bn1, netstat, ss, arp)

Security layer function:
```rust
fn validate_command(cmd: &str) -> Result<()> {
    // Blacklist: rm -rf /, mkfs, dd, reboot, halt, shutdown, init
    // Whitelist for exec_readonly: known read-only commands
    // exec allows anything not in blacklist (with confirmation)
}
```

- [ ] **Step 2: Add network actions**

- `scan_network` → Run `arp -a` or `ip neigh`, parse output, return device list
- `check_port` → `tokio::net::TcpStream::connect_timeout(addr, Duration::from_secs(5))`, return open/closed
- `network_interfaces` → Parse output of `ip addr` or `ifconfig`, return interface list with IPs

- [ ] **Step 3: Add hardware/system info actions**

- `resource_usage` → Read `/proc/meminfo`, `/proc/loadavg`, `df` output, return structured CPU/mem/disk
- `list_usb` → Run `lsusb` or parse `/sys/bus/usb/devices/`, return device list
- `system_info` → Collect hostname, os, arch, uptime, rustc version, disk free
